1. Library Management System

A library manages books, members, and loans.

Entities

  • Book (book ID, title, author)
  • Member (member ID, name, membership date)
  • Loan (loan ID, loan date, return date)

Relationships

  • A member borrows multiple books.
  • A book can be borrowed by multiple members over time.
  • Each loan is tied to one member and one book.

Design Decisions

  • Weak Entities: No weak entities were introduced since all entities can be uniquely identified.
  • Relationships: Established many-to-many relationships between members and books through loans.

Crow's Foot ER Diagram (Mermaid)

2. Restaurant Reservation System

A restaurant manages reservations, customers, and tables.

Entities

  • Customer (customer ID, name, contact number)
  • Reservation (reservation ID, date, time)
  • Table (table number, seating capacity, location)

Relationships

  • A customer can make multiple reservations.
  • A table can have multiple reservations over time.
  • Each reservation links one customer to one table.

Design Decisions

  • Composite Attributes: None were needed, as each attribute is atomic.
  • Relationships: Many-to-one relationships were established between reservations and both customers and tables.

Crow's Foot ER Diagram (Mermaid)

erDiagram Customer { string CustomerID string Name string ContactNumber } Reservation { string ReservationID date Date time Time } Table { string TableNumber int SeatingCapacity string Location } Customer ||--o{ Reservation : "books" Table ||--o{ Reservation : "is for" Reservation }o--|| Customer : "by" Reservation }o--|| Table : "for"

3. Real Estate Listing System

A real estate agency tracks properties, agents, and clients.

Entities

  • Property (property ID, address, listing price)
  • Agent (agent ID, name, contact information)
  • Client (client ID, name, budget)

Relationships

  • One agent lists multiple properties.
  • A property is listed by only one agent.
  • Clients can be interested in multiple properties.
  • A property can have multiple interested clients.

Design Decisions

  • Multivalued Attributes: No multivalued attributes were necessary, and each attribute is kept simple.
  • Weak Entities: There were no weak entities since properties, agents, and clients can be uniquely identified by their IDs.

Crow's Foot ER Diagram (Mermaid)

erDiagram Property { string PropertyID string Address float ListingPrice } Agent { string AgentID string Name string ContactInfo } Client { string ClientID string Name float Budget } Agent ||--o{ Property : "lists" Client ||--o{ Interest : "expresses" Property ||--o{ Interest : "gains" Interest }o--|| Client : "for" Interest }o--|| Property : "for"
